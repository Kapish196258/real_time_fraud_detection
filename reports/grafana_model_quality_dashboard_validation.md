# Grafana Model Quality Dashboard Validation Report

## 1. Purpose

This report documents the functional validation of the **Fraud Model Quality**
Grafana dashboard developed for the Real-Time Fraud Detection project.

The dashboard evaluates the quality of the Random Forest fraud classification
model using live predictions processed through the following pipeline:

```text
PaySim dataset
        ↓
Python Kafka producer
        ↓
Apache Kafka
        ↓
Spark Structured Streaming
        ↓
Random Forest fraud prediction
        ↓
Prometheus model-quality metrics
        ↓
Grafana Model Quality dashboard
```

The dashboard provides visibility into the model's confusion-matrix outcomes
and derived classification metrics.

## 9. Classification Metric Calculations

### 9.1 Precision

Precision measures how many transactions predicted as fraud were actually
fraudulent.

```text
Precision = TP / (TP + FP)
```

For the validated batch:

```text
Precision = 2 / (2 + 0)
          = 1.00
          = 100%
```

### 9.2 Recall

Recall measures how many actual fraud transactions were correctly detected.

```text
Recall = TP / (TP + FN)
```

For the validated batch:

```text
Recall = 2 / (2 + 0)
       = 1.00
       = 100%
```

### 9.3 F1 Score

The F1 score is the harmonic mean of precision and recall.

```text
F1 = 2 × TP / (2 × TP + FP + FN)
```

For the validated batch:

```text
F1 = (2 × 2) / ((2 × 2) + 0 + 0)
   = 1.00
   = 100%
```

### 9.4 Accuracy

Accuracy measures the proportion of all predictions classified correctly.

```text
Accuracy = (TP + TN) / (TP + TN + FP + FN)
```

For the validated batch:

```text
Accuracy = (2 + 1997) / (2 + 1997 + 0 + 0)
         = 1999 / 1999
         = 1.00
         = 100%
```

## 12. Interpretation of Results

The validated transaction range produced perfect classification results for
the tested batch:

- both actual fraud transactions were detected;
- no legitimate transaction was incorrectly classified as fraud;
- no fraud transaction was missed;
- all normal transactions were classified correctly.

The most important operational result is:

```text
False negatives = 0
```

A false negative represents an actual fraudulent transaction that the model
incorrectly predicts as normal.

Although the validated batch produced 100% values, the result must be
interpreted within the context of the tested transaction range. It does not
prove that the model will achieve 100% performance on every unseen dataset or
production workload.

The PaySim dataset is highly imbalanced, so recall, precision, F1 score, and
false-negative count provide more useful fraud-detection evidence than
accuracy alone.

## 15. Limitations

The current validation has the following limitations:

- the tested range contained only two actual fraud transactions;
- the dashboard counters are cumulative for the lifetime of the Spark process;
- restarting the metrics process resets in-memory Prometheus counters;
- perfect batch performance does not guarantee perfect production performance;
- the PaySim dataset is simulated rather than real banking data;
- no automatic model retraining is implemented;
- no concept-drift detection is currently included.
